# PRD: 聊天中实时编辑模型参数

## 简介

目前，右上角的聊天设置面板（模型选择、系统提示词、温度、上下文长度等）仅在发送第一条消息之前可用。一旦聊天开始，`ChatSettings` 组件被卸载，用户完全无法调整参数。本功能移除该限制，允许用户在对话的任何阶段修改全部模型参数。修改后的参数仅影响后续消息，并自动保存到数据库。

## 目标

- 允许用户在对话过程中随时访问和编辑聊天设置
- 参数变更自动保存到数据库，刷新页面或重新打开聊天后仍然生效
- 修改后的参数仅影响后续发送的消息，不影响已有消息
- 采用最小可用版本方案，复用现有设置 UI，改动最小化

## 用户故事

### US-001: 聊天进行中显示设置按钮

**描述：** 作为用户，我希望在发送消息后，右上角仍然能看到聊天设置按钮（模型名称 + 齿轮图标），以便随时进入设置。

**验收标准：**
- [ ] 当 `chatMessages.length > 0` 时，`ChatSettings` 组件仍然在聊天头部渲染
- [ ] 按钮显示当前模型名称和齿轮图标，样式与聊天开始前一致
- [ ] 点击按钮打开相同的设置弹窗
- [ ] 设置弹窗不会遮挡聊天消息区域
- [ ] 类型检查通过（`npm run type-check`）
- [ ] 在浏览器中验证 UI 效果

### US-002: 聊天中编辑全部参数

**描述：** 作为用户，我希望在对话过程中修改系统提示词、模型、温度、上下文长度等设置，以便在不重新开始对话的情况下微调参数。

**验收标准：**
- [ ] 聊天中可编辑全部参数：模型、系统提示词（Prompt）、温度（Temperature）、上下文长度（Context Length）、包含个人资料上下文、包含工作区指令、嵌入向量提供商
- [ ] 编辑任何参数后，内存中的 `chatSettings` 状态立即更新
- [ ] 设置表单的行为与聊天开始前完全一致（相同的验证、滑块、下拉框）
- [ ] 切换模型时，温度和上下文长度自动限制在新模型的有效范围内
- [ ] 类型检查通过（`npm run type-check`）
- [ ] 在浏览器中验证 UI 效果

### US-003: 自动保存设置变更到数据库

**描述：** 作为用户，我希望参数变更能自动保存到聊天记录中，以便刷新页面或重新进入聊天时保留最新设置。

**验收标准：**
- [ ] 在活跃聊天中修改参数时，调用 `updateChat()` 将更新写入数据库
- [ ] `chats` 表中以下字段被更新：`model`、`prompt`、`temperature`、`context_length`、`include_profile_context`、`include_workspace_instructions`、`embeddings_provider`
- [ ] 保存时刷新 `updated_at` 时间戳
- [ ] 刷新页面后重新打开聊天，显示最后保存的参数值
- [ ] 类型检查通过（`npm run type-check`）

### US-004: 修改后的设置仅影响后续消息

**描述：** 作为用户，我期望在对话中修改设置后，仅影响之后发送的新消息，已有的对话历史保持不变。

**验收标准：**
- [ ] 切换模型后，下一条消息使用新模型生成
- [ ] 修改系统提示词后，下一次 API 调用使用更新后的系统提示词
- [ ] 修改温度后，下一次 API 调用使用新的温度值
- [ ] 聊天历史中的已有消息不会被重新生成或修改
- [ ] 类型检查通过（`npm run type-check`）

## 功能需求

- FR-1: 无论 `chatMessages.length` 是否为 0，`ChatSettings` 组件都必须在聊天 UI 头部渲染
- FR-2: 所有设置字段（模型、系统提示词、温度、上下文长度、包含个人资料上下文、包含工作区指令、嵌入向量提供商）在活跃聊天中必须保持可交互
- FR-3: 在活跃聊天中（`selectedChat` 存在时）修改任何设置，系统必须调用 `updateChat()` 将变更持久化到 `chats` 表
- FR-4: 内存中的 `chatSettings` 状态是下一条消息 API 调用参数的唯一数据源
- FR-5: 聊天设置按钮必须显示当前模型名称，切换模型后动态更新
- FR-6: 设置变更不得触发消息重新生成或修改已有聊天历史

## 非目标（不在范围内）

- 不提供参数变更历史或撤销功能
- 不在消息流中显示"设置已变更"的视觉标记
- 不支持消息级别的参数覆盖（设置是聊天级别的，不是消息级别的）
- 不弹出确认对话框
- 不支持跨多个聊天批量修改设置

## 技术方案

### 需要修改的文件

| 文件 | 改动内容 |
|------|----------|
| `app/[locale]/[workspaceid]/chat/page.tsx` | 移除 `chatMessages.length === 0` 的条件渲染限制 |
| `components/chat/chat-ui.tsx` | 在聊天头部区域添加 `ChatSettings` 组件 |
| `components/chat/chat-settings.tsx` | 添加自动保存逻辑：当 `selectedChat` 存在时，设置变更后调用 `updateChat()` |
| `db/chats.ts` | 现有 `updateChat()` 已支持部分更新，无需修改 |

### 架构说明

- `ChatSettings` 组件目前仅在"空聊天"视图中渲染（`chat/page.tsx` 第 29-40 行）。需要在 `ChatUI` 组件中也渲染该组件
- `ChatbotUIContext` 中的 `chatSettings` 状态已被 `handleSendMessage()` 用于构建 API 请求参数，因此只需更新状态即可影响后续消息
- `db/chats.ts` 中的 `updateChat()` 接受 `TablesUpdate<"chats">` 类型，支持部分更新——无需新增数据库函数
- 无需数据库迁移，`chats` 表已包含全部所需字段

### 自动保存策略

- 使用防抖保存（300-500ms），避免拖动滑块时频繁写库
- 保存条件：`chatSettings` 值发生变化 **且** `selectedChat` 不为 null（即聊天已存在于数据库中）
- 对于新聊天（第一条消息发送前），保留现有行为：设置随第一条消息通过 `handleCreateChat()` 一起保存

## 成功指标

- 用户可以在对话的任何阶段修改全部聊天参数
- 设置变更在页面刷新后仍然保留
- 现有聊天功能无回归（消息发送、流式响应、检索增强）
